# Part 2: Custom App, Chat & Explore UI

Part 1 got us from raw text to a knowledge graph. This part covers what happened next — adding an AI chat interface, building explore pages, and eventually ripping out the framework that was slowing us down. All through the same Claude Code conversation.

---

## Prompt 6: Add a chat interface

> *"Now I want to set up a chat interface with the project. Something like this: https://github.com/nuxt-ui-templates/chat. I can add an Anthropic key to be able to use the Anthropic API."*

**What happened:** Claude examined the Nuxt UI chat template, understood the architecture (Vercel AI SDK, streaming, chat components), and built a custom implementation. When the Anthropic API key had no credits, we pivoted:

> *"Can we set up a local chat instead, with LM Studio server?"*

Claude rewired the chat to use LM Studio's OpenAI-compatible API at `localhost:1234`. No API costs. No cloud dependency. The chat connected to whatever model was loaded in LM Studio.

**Founder lesson:** Don't get blocked by billing or vendor issues. Claude can pivot the implementation in seconds. The architecture stayed the same — only the endpoint changed.

## Prompt 7: Make it look good

> *"The markdown on the page — I see the chat input, but the slide on the right doesn't slide out correctly."*

After debugging the Docus built-in assistant (which had broken dependencies), we built our own:

> *"Can we just make our own component?"*

**What happened:** Claude built a slide-out chat panel from scratch — a fixed-position aside that transitions in from the right, pushes the content over, and includes a header with close/expand/clear buttons, a message list, an input area, and a floating trigger button.

The streaming wasn't rendering at first (`v-html` doesn't parse markdown), so we brought in [Comark](https://comark.dev/):

> *"I want the Markdown. Can we use this? https://comark.dev/"*

Claude installed `@comark/nuxt` and wired it up for real-time streaming markdown rendering — bold, italics, block quotes, links, all appearing live as tokens arrive.

**Founder lesson:** Link Claude to what you want. Paste a URL, show a screenshot, reference a library. The more concrete the reference, the faster it gets built right.

## Prompt 8: Handle thinking tokens

> *"The thinking comes through as if it were the response. Can we collect that into a collapsed accordion or something, just like I get from Claude Code inside VS Code?"*

**What happened:** Claude separated `reasoning_content` tokens from `content` tokens in the SSE stream. During streaming, thinking shows in a compact auto-scrolling box with a pulsing brain icon. After completion, it collapses into an expandable `<details>` element — click to see the reasoning, or ignore it.

**Founder lesson:** Reference existing UX you like. "Like Claude Code's thinking blocks" gave Claude an exact target to match. Don't describe abstract requirements when you can point to concrete examples.

## Prompt 9: Connect the knowledge graph to the chat

> *"I don't know that our chatbot is really searching our docs, is it? We've created this knowledge graph — are we using it?"*

The honest answer was no. The chat was just forwarding questions to the LLM with a system prompt. No knowledge graph context.

> *"The built-in MCP is great."*

**What happened:** Claude built a knowledge graph search engine (`server/utils/search-graph.ts`) that loads the 66 metadata JSON files into memory and scores chapters by relevance:

- People matches score 4 points
- Topics and locations score 3 points each
- Events score 2 points
- Direct chapter references (like "Psalm 23") score 100 — always top result

The chat server route now searches the graph before every LLM call, injecting the matching chapters' metadata as context. Ask "Who was King David?" and the LLM gets the top 8 David-related chapters with their people, topics, events, and cross-references.

**Founder lesson:** Build the data layer first, then connect it to AI. An LLM without context hallucinates. An LLM with structured context from your own data gives grounded, linked answers.

## Prompt 10: Page-aware chat

> *"When we open the AI chat, can we feed the current page into the context automatically, as well as the graph of metadata behind it? That seems like a reasonable assumption that the user is wanting to chat about the current page."*

**What happened:** The chat panel now sends the current route path with every request. The server looks up that chapter's full metadata and injects it as "Currently viewing" context before the search results. If you're on Genesis 1 and ask "what does this mean?", the LLM knows exactly which chapter you're asking about.

**Founder lesson:** Think about what context the user implicitly carries. They're on a page — that's context. They just searched for something — that's context. Feed it to the AI automatically instead of making the user repeat themselves.

## Prompt 11: Remove the framework

> *"Can we just structure the project like a regular Nuxt project? Let's remove Docus — it doesn't feel fit for purpose."*

Docus had been causing friction throughout: the `content.config.ts` jiti crash, the broken AI assistant, CJS/ESM conflicts with the `extend` package. Every workaround added complexity.

**What happened:** Claude removed Docus, added `@nuxt/ui`, `@nuxt/content`, and `tailwindcss` as direct dependencies, created custom layouts, a docs page template, and an app shell. The first attempt crashed — "IPC connection closed" on every page.

After systematic debugging (stripping modules one by one, testing a fresh project), Claude found the root cause:

> Nuxt UI v4 requires `tailwindcss` as a direct dependency. Without it, the `@import "tailwindcss"` directive crashes Vite's SSR worker silently.

One `pnpm add tailwindcss` later, everything worked.

**Founder lesson:** When a framework fights you more than it helps, remove it. The Nuxt UI components (Header, Sidebar, Page, ContentNavigation, ContentSearch) give you everything Docus provided, without the abstraction layer getting in the way. And when things break — strip down to the minimum, add back one piece at a time, find the exact cause.

## Prompt 12: Build the explore pages

> *"We spoke about creating some explore pages, where we could look at themes, or people or something like that, right?"*

The pages were already built from the plan — but they weren't linked in the UI.

> *"I don't see links to them in the app, how does a user get there?"*

**What happened:** Claude added explore links to the sidebar (People, Topics, Locations, Eras with icons) above the Bible navigation, plus the header nav. Then added live-filtering search inputs to each explore index page:

> *"Let's create a search function on each of the explore subpages that live filters what is shown."*

Each page got a `UInput` with an icon that filters the list instantly as you type.

**Founder lesson:** Features don't exist until users can reach them. Always ask "how does a user get here?" after building something. And adding polish like live filtering is trivial when you can just ask for it.

## Prompt 13: Make the sidebar collapsible

> *"Can we have the sidebar width be changeable from the user? With the drag handle?"*

And then:

> *"Can we also make it collapsible or auto-expanding over the content? You know what I mean?"*

**What happened:** Claude switched from the basic `UPageAside` to `USidebar` with `collapsible="offcanvas"` and `rail` — the sidebar slides completely off-screen when collapsed, content fills the full width, and a thin rail on the edge lets you bring it back with a click.

**Founder lesson:** You don't need to know the component API. Describe the UX you want in plain language. Claude knows the component library and picks the right props.

## The meta lesson

This entire project — 1,189 annotated chapters, knowledge graph search, AI chat with streaming markdown and thinking blocks, explore pages with live filtering, collapsible sidebar, custom Nuxt UI app — was built in a single Claude Code conversation.

No design doc. No sprint planning. No Jira tickets. Just a founder describing what they want, and an AI building it in real time.

The prompts in this walkthrough are the actual prompts from the session. They're informal, sometimes vague, sometimes a URL or a screenshot. That's the point. You don't need to write specifications. You need to communicate intent.

**For founders:** the bottleneck is no longer engineering capacity. It's knowing what to build and being able to articulate it — even roughly. Claude Code handles the rest.
