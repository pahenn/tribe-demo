# Part 4: Agentic Chat

The chat worked. It had page context. It had knowledge graph search results injected into the prompt. But it was still a single-shot: the server searched once, injected the results, and sent everything to the LLM. The LLM couldn't ask follow-up questions to the data. It couldn't decide to dig deeper.

This part covers turning the chat into an agent.

---

## Prompt 19: Give it tools

> *"Now, are we able to create like a mini agent for our chat window? So, we ask a question, but pass the LLM a tool to be able to use our search API to dig deeper?"*

**What happened:** Claude rewrote the chat server route from a single LLM call into an agent loop. The LLM now receives four tools:

**`search_bible`** — Search the knowledge graph by keywords. Matches against people, topics, locations, events. Returns the top matching chapters with their full metadata.

**`get_chapter`** — Look up a specific chapter by path. Returns all metadata: people (with roles), topics, locations, events, era, key verses, and cross-references.

**`find_person_chapters`** — Find every chapter where a person appears. "Where is Moses mentioned?" returns 100+ chapters with paths.

**`find_topic_chapters`** — Find every chapter tagged with a topic. "Which chapters discuss covenant?" returns all matches.

### The agent loop

The server runs up to 5 rounds:

1. Send the user's question + tools to LM Studio
2. If the LLM returns tool calls → execute them against the knowledge graph
3. Feed the tool results back to the LLM
4. Repeat until the LLM generates a text response (or 5 rounds)
5. Stream the final text response to the browser

During tool execution, the thinking block shows `[Searching: search_bible]` so the user sees the agent working.

### What this enables

Ask "How is Moses connected to the exodus?" and the agent might:

1. Call `find_person_chapters("Moses")` — finds 100+ chapters
2. Call `find_topic_chapters("exodus")` — finds all exodus-themed chapters
3. Call `get_chapter("/docs/old-testament/exodus/chapter-14")` — gets the Red Sea crossing details
4. Synthesize a grounded answer with specific chapter links, cross-references, and quoted metadata

The LLM decides what to search for. It decides when it has enough information. It decides when to stop and answer. That's the difference between a chatbot and an agent.

### How it works technically

```
Browser → POST /api/chat → Server
                              ↓
                    Build system prompt + tools
                              ↓
                    Send to LM Studio (streaming)
                              ↓
                    ┌─ Text response? → Stream to browser, done
                    │
                    └─ Tool calls? → Execute against knowledge graph
                                     ↓
                              Add tool results to conversation
                                     ↓
                              Send again to LM Studio (round 2)
                                     ↓
                              ┌─ Text? → Stream, done
                              └─ More tools? → Execute, repeat
                                              (up to 5 rounds)
```

The key insight: the LLM sees the tool results as part of the conversation. Each round builds on the previous one. By round 3, the LLM has a rich picture assembled from multiple knowledge graph queries.

### Requirements

The model loaded in LM Studio must support function/tool calling. Most instruct models do — Llama 3.1+, Qwen, Gemma, Mistral. If the model doesn't support tools, it falls back to answering directly from its training data (no tool calls, no knowledge graph lookup).

**Founder lesson:** Giving an LLM tools is the difference between "answer from what you know" and "go find out." The tools are simple functions — they wrap the same search utilities we already built for the explore pages. The agent loop is ~50 lines of code. But the capability jump is enormous: the chat goes from a static Q&A to a research assistant that can traverse your data.

## The full architecture

At this point, the project has four layers:

**Layer 1: Content** — 1,189 markdown files with YAML frontmatter containing structured metadata (people, topics, locations, events, eras, key verses, cross-references).

**Layer 2: Index** — 66 JSON metadata files loaded into an in-memory index at startup. Supports keyword search, direct chapter lookup, and aggregation by any dimension.

**Layer 3: UI** — Browse (sidebar navigation), Explore (people/topics/locations/eras pages with filtering), Visualize (D3 force graph with click-to-expand).

**Layer 4: Agent** — LLM with tool access to the index. Can search, look up chapters, find people/topics, and synthesize grounded answers with links.

Each layer builds on the previous one. The agent uses the same functions as the explore pages. The explore pages use the same index as the search. The search indexes the same metadata that's in the frontmatter. It's the same data, surfaced four different ways.
