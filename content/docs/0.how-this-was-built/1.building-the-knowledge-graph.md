# Part 1: Building the Knowledge Graph

This entire project was built in a single Claude Code session. One conversation. No team of developers. No weeks of planning. Just a founder with an idea and an AI coding partner.

The goal: take the King James Bible — 66 books of raw markdown — and turn it into a structured, searchable, AI-queryable knowledge graph. Every chapter annotated with people, places, topics, events, and cross-references. Navigable. Explorable. Conversational.

Here's exactly how it happened, prompt by prompt.

---

## The starting point

A fresh Docus project with the KJV Bible dumped into the `content/` directory as 66 markdown files — one per book. Each file had all chapters jammed together under `## Chapter N` headings. No metadata. No structure. Just text.

## Prompt 1: Orient the AI

> *"Familiarize yourself with the repo. This is a nuxt content repo, and relies on nuxt ui for the ui. I have the generic docus starter here, but also in the content directory, I have the King James Bible in markdown. Clean up this repo so that it only has the bible material. Organize it appropriately."*

**What happened:** Claude read the entire repo structure, identified which files were Docus boilerplate vs. actual Bible content, removed the template docs, and reorganized the Bible into `Old Testament` and `New Testament` sections.

**Founder lesson:** Start by giving Claude the full picture. Don't jump to code. Let it understand what exists and what you want. The better the orientation, the better every subsequent prompt lands.

## Prompt 2: Define the vision

> *"I want to transform this into a detailed knowledge graph. Let's start by creating some rich frontmatter with topics, tags, labels, people, etc."*

Claude asked clarifying questions:
- What fields matter most?
- How granular — book level or chapter level?
- Should it be navigable in the UI?

The answers:

> *"I think probably the chapter level. Feel free to restructure the repo to accommodate this if it makes sense."*
> *"Yes, navigable please."*

**What happened:** Claude designed a complete plan — restructure each book into a directory, split each chapter into its own file, add YAML frontmatter with a rich schema (people with roles, topics, locations, era, events, key verses, cross-references). Then it wrote a Node.js script and executed it, producing 1,189 individual chapter files plus 66 book indexes.

**Founder lesson:** State the outcome you want, not the implementation. "I want a knowledge graph" is better than "add frontmatter to each file." Let the AI figure out the how — it often finds an approach you wouldn't have thought of.

## Prompt 3: Generate metadata at scale

> *"Use as many parallel agents as you can to speed up the process, since the chapters are all broken down."*

**What happened:** Claude launched 8 parallel agents, each responsible for a section of the Bible:

| Agent | Books | Chapters |
|-------|-------|----------|
| Pentateuch | Genesis – Deuteronomy | 187 |
| History | Joshua – Esther | 249 |
| Poetry & Wisdom | Job – Song of Solomon | 243 |
| Major Prophets | Isaiah – Daniel | 183 |
| Minor Prophets | Hosea – Malachi | 67 |
| Gospels & Acts | Matthew – Acts | 117 |
| Pauline Epistles | Romans – Philemon | 87 |
| General Epistles | Hebrews – Revelation | 56 |

Each agent read every chapter file, generated structured metadata based on the actual text, and wrote JSON files. A second script then merged the metadata into each chapter's YAML frontmatter. 1,189 chapters annotated. Zero manual editing.

**Founder lesson:** This is where AI shines for founders. A task that would take a biblical scholar weeks — reading and annotating 1,189 chapters — was done in minutes by running agents in parallel. The metadata isn't perfect, but it's 90% there and can be refined. Ship fast, iterate later.

## Prompt 4: Fix issues in real time

Along the way, bugs came up and were fixed immediately through conversation:

> *"Every book is duplicated in the search modal."*

Claude traced the issue to Nuxt Content indexing every heading as a separate search entry, then restructured the files to fix it.

> *"The chapters are in alphabetical order on the site, so it goes 1, 10, 11."*

Claude zero-padded the file prefixes (`001.chapter-1.md` instead of `1.chapter-1.md`).

**Founder lesson:** Don't plan for perfection. Build, test, report what's wrong, fix. The feedback loop with Claude Code is fast enough that iteration is cheaper than upfront planning.

## Prompt 5: Tell the story

> *"Write a blog post about what I'm doing here. Either as the landing page of all of this, or adjacent to that."*

**What happened:** Claude wrote the landing page and this how-it-was-built walkthrough — including the actual prompts from the conversation. Meta, but effective.

**Founder lesson:** Documentation is part of the product. Having Claude write it while the context is fresh means you get accurate docs that reflect what actually happened, not what you remember later.

## What we had at this point

- 66 books split into 1,189 chapter files
- Rich frontmatter on every chapter (people, topics, locations, events, era, key verses, cross-references)
- A working Docus site with sidebar navigation and search
- 66 metadata JSON files as the source of truth
- A landing page and documentation
- All scripts in `_artifacts/` for reproducibility

Total time: one Claude Code session. Total code written by hand: zero.

## For founders

If you're reading this thinking "I could do this with my own data" — you're right. The workflow:

1. Get your text into markdown
2. Tell Claude Code what it is and what you want it to become
3. Let it restructure, split, and organize
4. Design a metadata schema for your domain
5. Use parallel agents to annotate at scale
6. Build UI on top of the structured data

The scripts in `_artifacts/` are Bible-specific, but the pattern — orient, restructure, annotate, build — works for any corpus: legal documents, technical docs, research papers, product catalogs, customer support archives.

The competitive advantage isn't the code. It's the speed. What used to take a team weeks now takes a founder one conversation.
