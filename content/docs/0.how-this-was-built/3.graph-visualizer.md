# Part 3: Graph Visualizer & Dynamic UI

At this point we had 1,189 annotated chapters, explore pages for browsing by people/topics/locations/eras, and an AI chat. But the "graph" in knowledge graph was still invisible — the connections existed in the data but users couldn't see them. This part covers making the graph visual and making the UI context-aware.

---

## Prompt 14: Can we visualize it?

> *"Now, do we have enough info in our metadata to create a graph visualizer?"*

The answer was yes. Every chapter has:
- People connecting it to other chapters (Moses appears in 100+ chapters)
- Topics creating thematic clusters (all chapters about covenant)
- Locations forming geographic networks (everything mentioning Jerusalem)
- Cross-references as explicit directed edges between chapters

> *"Yes, build it."*

**What happened:** Claude installed D3.js and built three things:

**1. A graph data API** (`/api/graph/visualize`) that takes a filter type and value and returns nodes + edges. Five modes:
- `cross-references` — the web of explicit links between chapters
- `person` — star pattern: one person at center, all their chapters around them
- `topic` — same for a topic
- `location` — same for a location
- `chapter` — everything connected to one chapter (people, topics, locations, cross-refs)

**2. A D3 force-directed graph component** (`GraphVisualization.vue`) with:
- Physics simulation that settles into clusters
- Color-coded nodes: indigo (Old Testament), green (New Testament), amber (people), pink (topics), cyan (locations)
- Drag to rearrange, zoom and pan
- Hover for tooltip with navigation link
- Click to expand that node's graph
- Legend overlay

**3. A graph explorer page** (`/explore/graph`) with:
- Filter mode tabs (Cross-References, By Person, By Topic, By Location, By Chapter)
- Search input for finding specific entities
- Quick-pick preset buttons (Moses, Jesus, David, Jerusalem, Covenant, etc.)
- Live graph that updates as you change filters

**Founder lesson:** If your data has relationships, visualize them. A graph visualization turns abstract metadata into something tangible — users can literally see how the Bible connects to itself. And the implementation was one prompt: "yes, build it."

## Prompt 15: Click to explore, not navigate

> *"When I click a node in the graph visualizer, can it expand that graph of that node instead of navigating to that node?"*

**What happened:** One edit. Chapter nodes now expand to show their full graph (people, topics, locations, cross-references). Person/topic/location nodes expand to show all connected chapters. The graph becomes explorable by drilling into any node — you can follow connections through the data without ever leaving the page.

**Founder lesson:** The first implementation is rarely the right UX. The initial behavior (navigate away on click) was technically correct but broke the flow. The fix was a one-line change in intent, but it transformed the feature from a static view into an exploration tool.

## Prompt 16: Navigation links on hover

> *"On the hover label, can we have a link there that lets us navigate to it in the app?"*

**What happened:** The tooltip changed from a passive label to an interactive element with an "open →" link. Hover a node, then move to the tooltip to click through to that chapter/person/topic/location page. The tooltip stays visible with a 200ms grace period so you can reach it.

**Founder lesson:** Small UX details matter. The tooltip link means users can explore the graph visually AND drill into any node for the full page — best of both worlds.

## Prompt 17: Dynamic chat suggestions

> *"Can we actually make what pops up in the chat window for suggestions dynamic based on the graph of the current page?"*

Instead of showing the same four generic questions ("What is the Sermon on the Mount?", "Who was King David?") on every page, the chat panel now queries the current chapter's knowledge graph and generates contextual suggestions.

On Genesis 1, you see:
- "Explain this chapter to me"
- "Tell me about God in this chapter"
- "What does 'creation' mean in this context?"
- "How does this connect to John 1?"

On Romans 8, you might see:
- "Explain this chapter to me"
- "Tell me about Paul in this chapter"
- "What does 'justification' mean in this context?"
- "How does this connect to Galatians 3?"

**What happened:** The chat panel fetches `/api/graph/visualize?type=chapter&value={currentPath}` and builds suggestions from the returned people, topics, and cross-reference nodes.

**Founder lesson:** Context-aware UI is almost free when you have structured data. The knowledge graph isn't just for search — it can drive the entire UI. Every component can become smarter by querying the metadata.

## Prompt 18: Better header navigation

> *"I don't see links to the explore pages in the app, how does a user get there?"*

And then:

> *"Under Explore, can we add a link — maybe it's smaller across the bottom — to go to the explore home?"*

**What happened:** The header got dropdown menus using Nuxt UI's `NavigationMenu` component:
- **Bible** dropdown → Old Testament, New Testament
- **Explore** dropdown → People, Topics, Locations, Eras, Graph Visualizer, plus a footer link to the explore hub
- **How It Was Built** → direct link

The explore links were removed from the sidebar (since they're now in the header, accessible on every page).

**Founder lesson:** Features don't exist until users can reach them. After building the explore pages, we had to ask "how does a user get here?" three separate times before the navigation was right. Build the feature, then immediately think about the path to it.
